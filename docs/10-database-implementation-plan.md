# Pailo Database Implementation Plan

## Goal

Design a PostgreSQL database that is accurate, auditable, easy to query, and ready to grow from 15 initial shoe styles to a larger factory operation.

The database is the factory's operational memory. It must protect inventory correctness, task history, cost history, employee records, label versions, and production batch traceability.

## Target Database

- Amazon RDS for PostgreSQL.
- Upstream PostgreSQL current docs show 18.3 as current on 2026-05-02.
- Use PostgreSQL 18.x when available in the selected AWS region.
- If RDS does not yet support 18.x in the region, use the newest supported RDS PostgreSQL major version and plan a controlled upgrade.

## Database Principles

- Normalize core factory data.
- Use JSONB for flexible layout/template data, not as a replacement for relational modeling.
- Record every important change as a movement, event, status update, print job, or audit log.
- Avoid hard deletes for business records.
- Use clear human-readable codes plus internal IDs.
- Keep all timestamps in UTC.
- Store display currency as NPR.
- Store files in S3 and file metadata in PostgreSQL.
- Add indexes based on real query paths, not guesswork alone.

## Schemas

Recommended simple launch structure:

```text
public
```

Use prefixed tables instead of many schemas at launch. Add separate schemas later only if needed.

Potential future schemas:

```text
app       # operational tables
audit     # audit history
jobs      # background jobs/outbox
analytics # materialized reporting tables
```

## ID And Code Strategy

Use UUID primary keys generated by the application. Prefer UUIDv7 if the chosen Python library is mature; otherwise use UUIDv4 plus indexed timestamps.

Use human-readable codes for factory operations:

- Product style: `PAI-2026-SNK-001`.
- SKU: `PAI-2026-SNK-001-BLK-42`.
- Work order: `WO-2026-000001`.
- Task: `TASK-2026-000001`.
- Purchase order: `PO-2026-000001`.
- Supplier: `SUP-0001`.
- Employee: `EMP-0001`.

Codes should be unique and generated server-side.

## Core Table Groups

### Identity And Employees

Tables:

- `users`
- `roles`
- `permissions`
- `role_permissions`
- `employees`
- `employee_skills`
- `employee_documents`

Important indexes:

- `users(cognito_sub)` unique.
- `users(email)` unique where not null.
- `employees(employee_code)` unique.
- `employees(status, department)`.

### Task Management

Tables:

- `task_boards`
- `tasks`
- `task_status_updates`
- `task_comments`
- `task_templates`
- `task_dependencies`

Important columns on `tasks`:

- `status`
- `priority`
- `assigned_to_employee_id`
- `assigned_team`
- `work_order_id`
- `product_style_id`
- `material_id`
- `supplier_id`
- `due_at`
- `started_at`
- `completed_at`
- `reviewed_at`
- `blocked_reason`
- `version`

Important indexes:

- `tasks(status, due_at)`.
- `tasks(assigned_to_employee_id, status, due_at)`.
- `tasks(work_order_id, status)`.
- `tasks(board_id, status, priority)`.
- `task_status_updates(task_id, created_at desc)`.

Task status updates are append-only. Do not overwrite history.

### Products And Gallery

Tables:

- `product_styles`
- `product_colorways`
- `product_variants`
- `product_photos`
- `attachments`

Important indexes:

- `product_styles(style_code)` unique.
- `product_styles(status, category)`.
- `product_variants(sku)` unique.
- `product_variants(style_id, colorway_id, size)` unique.

For the initial 15 styles, seed product categories, size systems, and style statuses.

### BOM And Costing

Tables:

- `boms`
- `bom_items`
- `cost_snapshots`
- `price_lists`

Rules:

- BOM versions are immutable after approval.
- New material prices should not silently change old production cost snapshots.
- Work orders should store estimated cost snapshot at creation time.

### Suppliers And Purchasing

Tables:

- `suppliers`
- `supplier_contacts`
- `purchase_orders`
- `purchase_order_items`
- `supplier_price_history`
- `supplier_quality_events`

Important indexes:

- `suppliers(supplier_code)` unique.
- `purchase_orders(status, expected_delivery_date)`.
- `supplier_price_history(material_id, supplier_id, effective_date desc)`.

### Inventory

Tables:

- `materials`
- `material_batches`
- `inventory_locations`
- `inventory_movements`
- `finished_good_stock`
- `stock_counts`
- `stock_count_lines`

Rules:

- Inventory changes only through `inventory_movements`.
- Movement inserts and stock updates happen in the same transaction.
- Negative stock requires explicit manager override and audit reason.
- Material batch cost should support actual costing later.

Important indexes:

- `materials(material_code)` unique.
- `materials(category, active)`.
- `material_batches(material_id, quantity_available)`.
- `inventory_movements(item_type, item_id, created_at desc)`.
- `finished_good_stock(variant_id, quality_status, storage_location)`.
- `finished_good_stock(work_order_id)`.

### Production

Tables:

- `work_orders`
- `work_order_size_lines`
- `production_stages`
- `work_order_stage_logs`
- `material_reservations`

Important indexes:

- `work_orders(status, planned_start_date)`.
- `work_orders(work_order_code)` unique.
- `work_order_size_lines(work_order_id, size)` unique.
- `work_order_stage_logs(work_order_id, stage_id, created_at)`.

### Quality

Tables:

- `quality_inspections`
- `quality_defects`
- `rework_tasks`

Important indexes:

- `quality_inspections(work_order_id, inspected_at desc)`.
- `quality_defects(defect_type, severity)`.

### Labels

Tables:

- `label_templates`
- `label_template_versions`
- `label_print_jobs`
- `label_print_job_items`

Store template layout in JSONB:

```json
{
  "page": { "width_mm": 210, "height_mm": 297 },
  "label": { "width_mm": 70, "height_mm": 35 },
  "elements": [
    { "type": "text", "field": "brand_name", "x": 4, "y": 3 },
    { "type": "barcode", "field": "sku", "x": 4, "y": 20 }
  ]
}
```

Important rules:

- Never overwrite approved template versions.
- Print jobs store template version ID.
- Generated PDF location is stored as S3 key.

### Orders And Dispatch

## Constraints

Use database constraints where they protect the business:

- Unique style codes, SKUs, work order codes, employee codes, supplier codes.
- Positive quantity checks.
- Valid status checks through enums or lookup tables.
- Foreign keys for operational relationships.
- Non-null required ownership fields.
- Unique variant by style/color/size.

Do not rely only on application validation for inventory and production rules.

## Indexing Plan

Launch indexes should support:

- Dashboard queries.
- My Tasks query.
- Task board query.
- Product search.
- Inventory lookup.
- Work order status list.
- Movement history.
- Label print history.

Use `EXPLAIN ANALYZE` before adding complex indexes. Remove unused indexes if write performance becomes an issue.

## Search

Use PostgreSQL full-text search first.

Search targets:

- Product styles.
- Materials.
- Suppliers.
- Employees.
- Work orders.
- Tasks.

Do not add Elasticsearch/OpenSearch at launch. PostgreSQL search is enough for 15 styles and low traffic.

## Migrations

Use Alembic.

Migration rules:

- Every schema change is a migration.
- Migrations run in CI against a blank database.
- Migrations run in staging before production.
- Destructive migrations require backup and explicit approval.
- Seed data is versioned and repeatable.

Initial seed data:

- Roles and permissions.
- Production stages.
- Task statuses and board types.
- Material categories.
- Product categories.
- Label template sample recreated from `Sticker 42.doc`.
- Initial 15 product style placeholders.

## Backup And Recovery

For launch:

- RDS automated backups enabled.
- 7 to 14 day retention depending on budget.
- Daily snapshot for production if affordable.
- Test restore at least monthly.
- Store critical exports in S3 with lifecycle rules.

Recovery objectives for launch:

- RPO: 24 hours maximum, ideally under 1 hour with point-in-time restore.
- RTO: same business day for MVP.

When factory dependency increases, tighten these targets and move to Multi-AZ.

## Data Privacy

Employee records include sensitive personal data.

Rules:

- Restrict employee sensitive fields to HR/admin roles.
- Audit reads of sensitive documents if needed.
- Do not expose employee wage data to worker roles.
- Store documents in private S3 paths.
- Use signed URLs with short expiry.

## Performance Expectations

With 5 active users/day and 15 starting styles, PostgreSQL will be far from capacity. Still design correctly:

- Keep dashboard queries pre-aggregated or efficient.
- Use pagination.
- Keep task board payloads bounded by date/status.
- Avoid returning full audit history by default.
- Use indexes for status/date/assignee filters.

## Database Deliverables

- Alembic migration baseline.
- SQLAlchemy models.
- Seed data scripts.
- Factory status lookup tables.
- Inventory movement transaction functions/services.
- Indexes for MVP screens.
- Backup/restore runbook.
- Data dictionary exported from models or docs.
